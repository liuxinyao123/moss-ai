# MOSS AI 功能缺口与开发建议

基于当前代码与文档梳理的「还缺什么」清单，便于排期开发。

---

## 一、后端目前没有的能力

### 1. 对话/补全 API（高优先级）
- **现状**：后端没有「用户发一句话 → 返回 AI 回复」的接口。桌面端聊天依赖 **Electron 主进程** 读 `~/.openclaw/openclaw.json` 的 token，直接请求本机 OpenClaw Gateway（如 `http://127.0.0.1:18789/v1/chat/completions`）。
- **影响**：浏览器直接打开 `desktop/index.html` 或 `collaboration.html` 时，没有真实 AI 回复，只能走模拟文案；第三方/移动端也无法通过 3001 发起对话。
- **建议**：在 `backend/server.js` 增加例如 `POST /api/chat` 或代理到 OpenClaw 的 `POST /v1/chat/completions`，从请求体或 header 带 agentId、model、messages；token 可从同路径 `openclaw.json` 或环境变量读取，便于非 Electron 场景使用。

### 2. 用户认证（中高优先级）
- **现状**：所有 REST 与 WebSocket 均无鉴权，仅 WebSocket 的 `auth` 消息用 `agentId` 在库里存在性校验，无用户/登录概念。
- **影响**：谁都能调 API、连 WS，不适合多用户或对外暴露。
- **建议**：按需增加：登录/注册（或对接现有 SSO）、session 或 JWT、在 Express 里加鉴权中间件；WebSocket 可在首次 `auth` 时校验 token 再绑定 agentId。

### 3. Hub（定时任务 / 心跳）未接入后端（中优先级）
- **现状**：`hub/` 下有 Cron 定时任务、心跳监控、活动记录（`cron-scheduler.js`、`heartbeat.js`、`activity-store.js`），但 **backend/server.js 没有 require Hub**，也没有暴露「增删查定时任务」的 API。
- **影响**：定时任务、心跳、活动记录目前无法通过 3001 使用或配置。
- **建议**：要么在 backend 启动时初始化 Hub（传入与 server 一致的 workspace/DB 等），并增加如 `GET/POST /api/cron/jobs`、`GET /api/activity`；要么把「定时任务/心跳」做成独立进程，通过 HTTP 或消息队列与 3001 通信，由你架构选择。

## 二、前端/桌面缺口（来自 desktop/README「下一步」）

| 项 | 说明 |
|----|------|
| 协作页集成到 Electron | 当前协作界面主要用浏览器或单独页面打开，未作为 Electron 主窗口的一屏或 tab。 |
| 用户认证 | 前端无登录/登出，无 token 传递。 |
| Agent 配置编辑 | 列表与详情里没有「编辑 agent 的 skills/domains/availability」等表单与保存。 |
| 任务详情视图 | 协作任务只有列表/进度，缺少单任务详情（子任务、日志、结果展开）。 |
| 消息搜索 | 频道/协作消息没有按关键词或时间范围搜索。 |
| 数据导出 | 无导出智能体/任务/频道/记忆为 JSON/CSV 等。 |

## 三、技能系统缺口

### 1. 永久删除未实现（小）
- **位置**：`backend/skill-api.js`，`deleteSkill` 当 `permanent === 'true'` 时直接返回 501。
- **建议**：实现从 `skills/` 目录删除对应技能文件夹并从 registry 移除；若涉及 GitHub 安装的，可同时删 `community/github/xxx` 并更新 registry。

### 2. 技能组合/串联（设计有，代码无）
- **设计**：`skill-system-design.md` Phase 4 提到「技能组合：多个技能串联执行」。
- **现状**：执行器只做单技能执行，没有 pipeline/DAG。
- **建议**：先定义「组合技能」的配置格式（如顺序列表或 DAG），再在 skill-executor 或单独模块里做编排。

### 3. 技能结果自动写入记忆（设计有，未实现）
- **设计**：同文档「记忆系统集成 - 技能结果自动存储到记忆」。
- **现状**：技能执行结果没有自动调用 `memory-system.indexMemory`。
- **建议**：在技能执行成功回调里，按配置或默认策略把结果摘要写入记忆索引（可带 agentId、skillId 等 metadata）。

---

## 四、协作/多智能体缺口

### 1. 委托是否真的调 AI
- **现状**：SmartDelegator 拆任务、选 Agent、发消息；TaskCollaborator 管子任务状态。但 **backend 没有挂载 core/Engine**，委托消息若需要「由被委托 Agent 调 LLM 执行」则缺少执行端。
- **建议**：若希望委托后由 AI 真正干活，需要在后端接入 Engine（或等价地调用 OpenClaw），在「子任务分配给某 agent」时触发一次 LLM 调用并把结果回写；或明确当前委托只做「任务与消息流转」，执行由桌面/其他服务负责。

### 2. 能力矩阵与后端 agents 同步
- **现状**：AbilityMatrix 用文件存能力；agents 表在 SQLite，有 skills/domains 等字段。两处可能不一致。
- **建议**：创建/更新 agent 时同步写入 AbilityMatrix；或能力矩阵改为以 SQLite 为主，从 agents 表派生。

---

## 五、外部渠道与 Bridge（Telegram/QQ/飞书）

- **现状**：`lib/bridge/` 下有 Telegram、QQ、飞书适配器，依赖 **Hub** 的 `getActiveAgent()` 和 `engine.processMessage()`，而 Hub 未在 backend 中启动。
- **影响**：无法通过这些渠道收消息并自动用 AI 回复。
- **建议**：若需要这些渠道，要么把 Hub+Engine 接入 backend（或单独进程），由 Hub 统一调 Engine 并回写；要么在 backend 单独实现一套「webhook 入站 → 调 OpenClaw → 回写渠道」的轻量链路，与现有 Hub 解耦。

---

## 六、记忆与对话的衔接

- **现状**：记忆系统有索引、搜索、摘要；`/api/compile-memory` 是手动编译。桌面聊天产生的对话没有看到「自动写入 memory-system 索引」的逻辑。
- **建议**：若希望「对话即记忆」，可在后端 chat 接口（或桌面发送消息后）在成功回复后调用 `memory-system.indexMemory`，把 user+assistant 摘要或关键信息写入，便于后续语义搜索。

---

## 七、运维与可观测性

- **日志**：目前主要靠 console 和 `logs/backend.log`，没有按级别、请求 ID 的结构化日志。
- **配置**：端口、DB 路径、OpenClaw 地址等多为写死或从 workspace 推导，建议用 `.env` 或 `config.json` 统一管理，便于部署与多环境。
- **监控**：无 metrics（如请求延迟、技能执行耗时分布、WS 连接数），可按需加简单指标接口或 Prometheus 导出。

---

## 八、建议的优先级（按「先跑通再完善」）

1. **P0**：**后端对话 API**（`POST /api/chat` 或代理 OpenClaw）—— 否则非 Electron 场景无法真正对话。
2. **P1**：**用户认证**（登录 + token + 鉴权中间件）—— 若计划多用户或对外。
3. **P1**：**桌面：Agent 配置编辑 + 任务详情视图**—— 日常使用最直观。
4. **P2**：**Hub 接入或定时任务 API**—— 需要定时/心跳时再上。
5. **P2**：**技能永久删除 + 技能结果写记忆**—— 小功能，补全体验。
6. **P3**：技能组合、Bridge 接入、消息搜索、数据导出、可观测性。

如果你确定先做哪一块（例如「先做后端 chat 接口」或「先做认证」），可以按该优先级拆成具体任务列表和接口设计再实现。
